# Roadmap

Дорожная карта платформы раздачи. Что уже работает — и куда двигаемся дальше.
Статусы: ✅ сделано · 🟡 в работе · ⬜ запланировано.

Связанные документы: [`ARCHITECTURE.md`](ARCHITECTURE.md),
[`docs/BOARD.md`](docs/BOARD.md) (рабочая доска по агентам),
[`docs/MULTI_ENGINE.md`](docs/MULTI_ENGINE.md).

## Сделано

### Фаза 0 — База платформы ✅
- Многокомпонентный стек на Docker Compose: Postgres, Redis, `api`, `engine`, `queue_worker`, `web`.
- Чёткие границы модулей (`db` владеет схемой; `engine` — раздачей; `api` — оркестрацией).
- libtorrent-движок с внутренним HTTP API; mock-движок для тестов без libtorrent.
- Alembic-миграции, CI (pytest + ruff + сборка web).

### Фаза 0.5 — Multi-engine ✅
- 6 независимых движков `b1…b6`, по тому/диску и BitTorrent-порту на каждый.
- `engine_id` в БД, реестр `config/engines.ct400.json`, маршрутизация `EnginePool` по `save_path`.
- Override-композ для хранения на `/mnt/media` (CT 400).

### Фаза 1 — Реальное сидирование + паритет (базовый) ✅
- Сквозная проверка end-to-end: перенос 20 раздач с сидбокса, хэш-чек, анонс, отдача пирами.
- per-torrent fastresume (быстрый рестарт без рехэша), сохранение `.torrent` и session.state.
- Список файлов + приоритеты, список трекеров + reannounce, force recheck, per-torrent лимиты.
- Обогащённый runtime: size / downloaded / ratio / seeds / ETA / added / лимиты.

### Фаза 2 — Паритет с важными функциями ruTorrent ✅
- Добавление по magnet / `.torrent`-файлу / **URL на `.torrent`**.
- Метки/категории (БД + API + UI), фильтр по метке.
- Глобальная статистика сессии и глобальные лимиты (агрегация по всем движкам).
- Список: поиск, сортировка, фильтр по статусу/метке, мультивыбор и **bulk** (пауза/старт/удаление).
- Трекеры: добавление/удаление на раздачу.
- **Корневой фикс restore-pause:** `auto_managed=False` + неограниченные `active_*`;
  честный `status_from_runtime`; авто-resume сидов при restore. Проверено: 21/21 `seeding`.

### Фаза 2.6 — Самовосстановление движка ✅
- Движок при старте сам поднимает раздачи со своего тома (fastresume/.torrents),
  без участия API. Перезапуск только движков больше не оставляет их пустыми
  (`SEEDING_ENGINE_SELF_RESTORE`, по умолчанию вкл.).
- Фикс репортинга `listening_port` per-engine в `session/stats`.

### Фаза 2.7 — Веб-UX ✅
- Отдельная страница «Настройки» (`#/settings`): глобальные лимиты сессии вынесены с главной.
- Добавление торрента через модальное окно (кнопка «+ Добавить торрент» в шапке);
  инлайн-панель убрана с главной.

### Фаза 2.8 — Мульти-загрузка `.torrent` ✅
- Бэкенд: `POST /api/v1/torrents/upload-batch` — multipart со списком файлов и общими
  `save_path`/`label`. Частичный успех: сбой одного файла не отменяет остальные,
  «осиротевшие» строки в БД откатываются при отказе движка. Отчёт по каждому файлу
  (`{total, ok, failed, items[]}`).
- UI: множественный выбор в модалке (`multiple`), один файл — как раньше (с названием),
  несколько — батч с итоговым тостом и перечислением ошибочных файлов.

### Фаза 2.9 — Выбор движка при добавлении ✅
- UI: вместо ручного ввода пути — выпадающий список движков со свободным местом
  (`своб. X из Y`) и пометкой офлайн; «Дополнительно: свой путь» свёрнут и переопределяет
  выбор движка при необходимости.
- Бэкенд: `engine_id` в `create`/`upload`/`upload-batch`/`url` — путь берётся из
  `storage_prefix` движка (обратная совместимость по `save_path` сохранена).
- Движок отдаёт `disk_total`/`disk_free` (`shutil.disk_usage` тома `/data`) в `session/stats`;
  `/api/v1/engines` обогащён местом на диске и `online`.

### Фаза 3.1 — Массовые метки ✅
- Бэкенд: `POST /api/v1/torrents/bulk/label` (`{ids, label}`, пустая строка — снять метку).
- UI: поле «Метка для выбранных» в тулбаре с подсказками-шаблонами (`datalist` из
  существующих меток `/labels`), назначение по Enter/кнопке, обновление фильтра меток.

### Фаза 3.2 — UX списка ✅
- Сохранение фильтров/сортировки/вида в `localStorage` (восстанавливаются при перезагрузке),
  кнопка «Сброс фильтров».
- Пресет вида списка: «строки» (уплотнённые ряды) / «плитки» (адаптивная сетка карточек).
- Явный переключатель темы (как в системе / светлая / тёмная) на странице «Настройки»,
  через `data-theme` на `<html>`, сохраняется в браузере.

### Фаза 3.3 — Push-обновления (SSE) ✅
- Бэкенд: `GET /api/v1/stream` (text/event-stream) шлёт снапшоты `{torrents, stats}` каждые
  N секунд (`interval`, 1–30). Авторизация query-параметром `api_key` (EventSource не шлёт
  заголовки). Учитывает `request.is_disconnected()`.
- nginx: отдельный `location = /api/v1/stream` с `proxy_buffering off`, длинным
  `proxy_read_timeout` и keep-alive.
- Web: `EventSource` на странице списка вытесняет поллинг; при ошибке/буферизации —
  автоматический откат на таймерный поллинг. На скрытой вкладке поток ставится на паузу.

### Фаза 3.4 — Десктоп-GUI (PySide6) ✅
- Подключение с API-ключом (`X-API-Key`), URL и ключ сохраняются в конфиг.
- Сводка сессии (раздачи/активные, ↓/↑, отдано, онлайн-движки) и фоновое автообновление
  списка (не морозит UI).
- Тулбар: поиск по имени/метке/hash, фильтр по статусу, сортировка.
- Модалка добавления: вкладки Magnet / URL / Файлы (**мультивыбор** `.torrent`), выбор
  движка со свободным местом, метка с подсказками, необязательный свой путь.
- Таблица (id/имя/метка/статус/%/размер/↓/↑/сиды-пиры/ratio/ETA/движок), мультивыбор и
  **массовые действия** (пауза/старт/метка/удаление с опцией удаления файлов).
- Детали по двойному клику: файлы и трекеры, recheck/reannounce.

### Фаза 3.5 — Надёжность ✅
- Graceful shutdown + устойчивость к падению: `stop()` сохраняет fastresume/session.state
  по SIGTERM; **периодическое сохранение** (`SEEDING_FASTRESUME_SAVE_INTERVAL`, 300s) на случай
  `kill -9`/паники; движкам выставлен `stop_grace_period: 60s`.
- Нормализация `save_path` на бэкенде: `normalize_save_path` (слэши/дубли/хвост) + строгий
  `match_engine_id` без молчаливого дефолта. `_resolve_target` отклоняет относительные пути и
  пути вне всех `storage_prefix` явной 422 (раньше торрент «уезжал» в дефолтный движок).
- Health/ретраи: `EngineClient` переподключается при сетевых сбоях
  (`httpx.AsyncHTTPTransport(retries=N)`, env `SEEDING_ENGINE_HTTP_RETRIES`, по умолчанию 2);
  `/api/v1/engines` отдаёт `online` по доступности движка.
- Бэкапы: `scripts/backup.sh` — `pg_dump` БД + метаданные раздач каждого движка
  (`.fastresume`/`.torrents`/`session.state`), ротация последних `KEEP` копий. Контент не
  копируется (восстанавливается ре-сидированием).

### Качество / эксплуатация ✅
- Web-фиксы вёрстки (стилизация URL-поля, панель действий списка, верхняя панель/чипы).
- Документация: ARCHITECTURE/ROADMAP, MULTI_ENGINE, INTEGRATION, QA-чеклист.

## Дальше

### Фаза 4 — Перенос раздач между движками ✅
Цель: переносить раздачу с одного движка на другой, в том числе когда движки на **разных
машинах**.

**Одна машина — сделано ✅** (проверено на CT400, реальный перенос b1→b2):
- ✅ Перемещение данных «pull через /media»: целевой движок копирует контент из общего
  read-only mount (`/media`) в свой том и перепроверяет хэш (`import-local`); `EngineSpec.media_path`
  задаёт, где контент движка виден другим.
- ✅ Оркестрация в API (`POST /torrents/{id}/migrate?engine_id=`): фоновый перенос, статус
  `migrating` (не перетирается рантаймом), атомарная смена `engine_id`/`save_path` только после
  подтверждённой полной копии; при неполной копии — откат (частичная копия на цели удаляется,
  источник остаётся), статусы корректно восстанавливаются.
- ✅ Бережно к раздаче: пауза источника → копия → recheck → seeding на цели; `.torrent`
  переносится, источник (контент + fastresume + .torrent) удаляется только после успеха.
- ✅ Web: кнопка «Перенести» в деталях с выбором целевого движка (свободное место, офлайн),
  статус «Перенос» с подсветкой.

**Между машинами — сделано ✅** (HTTP-транспорт через оркестратор, проверено локально b1→b2):
- ✅ Контент идёт по сети tar-потоком: движок-источник отдаёт `GET …/content`, оркестратор
  проксирует поток в приёмник (`stage-remote` + `POST …/import-remote` телом запроса), приёмник
  распаковывает (`tarfile` с `filter="data"` против path-traversal), затем add + recheck.
  Топология hub-and-spoke: достаточно связи «движок↔оркестратор», прямой доступ движок↔движок не нужен.
- ✅ Выбор транспорта `transport=auto|media|http` в `/migrate`: `auto` берёт `media` для общего
  `/media`, иначе сетевой `http`; движки без `media_path` теперь переносятся по сети, а не 422.
- ✅ Прогресс переноса в реальном времени: фазы `preparing → copying(%) → checking(%) → done/error`
  (движок считает байты копии/стрима, API агрегирует в `GET /torrents/{id}/migrate-status`),
  web показывает живой прогресс-бар вместо «вечного» бейджа. Плюс скорость переноса и ETA:
  оркестратор оценивает Б/с по дельте скопированных байт (окно ≥1с), web показывает
  `… · 180 МБ/с · ост. 3 с` на фазе копирования.

**Дальнейшие улучшения ⬜:**
- ✅ Прямой трансфер движок→движок (`transport=direct`) — оптимизация пропускной способности.
  Приёмник сам тянет контент у источника (`content-manifest` + `content-file` с Range,
  токен `X-Engine-Token` + проверка по `SEEDING_ENGINE_TLS_CA`), данные идут одним хопом
  мимо оркестратора (а не A→оркестратор→B, как в `http`). Оркестратор остаётся дирижёром:
  стартует, опрашивает прогресс у приёмника (`migrate-progress`), переключает БД, чистит
  источник; возобновляемость та же (пофайловая докачка). `auto` выбирает по факту:
  общий `/media` → `media`, иначе приёмник видит источник (`peer-check`) → `direct`,
  иначе → проксируемый `http`. Проверено на CT400: direct b1→b2 (данные напрямую по TLS),
  и устойчивость — SIGKILL приёмника на 16 % с последующим возобновлением до конца.
- ✅ Возобновление прерванного переноса. Состояние переноса хранится в БД
  (`migration_jobs`, миграция 0010) и переживает перезапуск оркестратора. При сбое
  частичная копия на цели НЕ удаляется — повтор докачивает только недостающее:
  для `media` копирование инкрементальное по файлам (пропуск файлов с совпадающим
  размером), для `http` — пофайловая докачка с Range (манифест + content-file/import-file
  + import-finalize). Источник остаётся нетронутым (на паузе) до подтверждённой полной
  копии. В UI у прерванного переноса — кнопки «Возобновить» / «Отменить» (полная отмена
  удаляет частичную копию и возвращает источник). Проверено на CT400: HTTP-перенос
  оборван на 70 %, после рестарта движка возобновлён до конца без копии с нуля.

### Фаза 4.5 — Распределённые движки и быстрый онбординг ✅
Цель: поднять движок на новой машине за минуты и быстро подключить к нашему оркестратору.
- ✅ Регистрация движка по **API-ключу**: движок при старте уходит в retry/heartbeat-цикл и сам
  регистрируется в оркестраторе (`POST /api/v1/engines/register`, заголовок `X-Register-Key`);
  id/url/`storage_prefix`/порт выводятся из env по соглашению — без ручной правки `engines.json`.
- ✅ Динамический реестр движков в БД (таблица `engines`, миграция 0004): `EnginePool` сливает
  статику и БД (`refresh` периодически + сразу после регистрации), пустые поля из БД не затирают
  статику (`media_path` сохраняется — перенос работает). Проверено на CT400: 6 движков
  саморегистрируются, перенос b2→b3 проходит через реестр из БД.
- ✅ Cross-machine перенос контента (HTTP-стрим через оркестратор) + recheck — см. Фазу 4.
- ✅ Авто-выбытие движков по `last_seen`: динамические движки без heartbeat дольше TTL
  (`SEEDING_ENGINE_TTL`, по умолчанию 180с = 3 пропуска) выбывают из активного пула на
  ближайшем refresh; статические из `engines.json` не выбывают. Полный реестр с
  `last_seen`/`stale`/`in_pool` — `GET /api/v1/engines/registry`. Проверено на CT400.
- ✅ «One-liner» развёртки движка на чистой машине: `docker-compose.engine.yml` +
  `.env.engine.example` + `scripts/deploy-engine.sh` (env: адрес оркестратора + API-ключ +
  BT-порт + advertise-URL). Движок собирается из репозитория, сам регистрируется и шлёт
  heartbeat. Гайд — `docs/REMOTE-ENGINE.md`.
- ✅ Безопасный транспорт между оркестратором и удалёнными движками:
  - Аутентификация: внутренний API (`/internal/v1/**`) защищён общим токеном
    `SEEDING_ENGINE_API_TOKEN` (заголовок `X-Engine-Token`, сравнение постоянного времени,
    без токена → 401; `/health` открыт).
  - Шифрование: TLS на внутреннем API (`SEEDING_ENGINE_TLS=1`), оркестратор проверяет
    серверный сертификат движка по приватному CA (`SEEDING_ENGINE_TLS_CA`). Генерация —
    `scripts/gen-certs.sh`, движок отдаёт `https://`, URL в реестре — https.
  - mTLS (клиентский сертификат оркестратора) — опционально: `SEEDING_ENGINE_MTLS=1`.
  - Проверено на CT400: TLS 1.3, https+CA+токен → 200, без токена → 401, plain-http к TLS-порту
    и https без CA отвергаются. Гайд — `docs/REMOTE-ENGINE.md`.
- ✅ Проверка связности при онбординге: `GET /api/v1/engines/{id}/connectivity` — оркестратор
  замеряет доступность внутреннего API (`reachable` + `api_latency_ms`) и тянет сетевой статус
  движка (`GET /internal/v1/net/status`): слушает ли libtorrent BT-порт (`listening_port`),
  был ли хоть один входящий коннект (`has_incoming` = признак, что порт открыт снаружи/NAT
  проброшен), число DHT-узлов и пиров.
  - Проверено на CT400: все 6 движков reachable по TLS (1–2 мс), слушают порты 50001–50006,
    DHT населён; `has_incoming=True` на b1/b3/b4/b6 подтверждает доступность BT-портов снаружи.

### Фаза 5 — Безопасность и многопользовательский режим 🟢
- ✅ Ролевой доступ по именованным API-ключам: таблица `api_keys` (хранится SHA-256, не сам
  ключ), роли `viewer < operator < admin`. Проверка по HTTP-методу (чтение → viewer,
  изменения → operator), управление ключами → admin. Ключи из env `SEEDING_API_KEYS`
  считаются admin; если ключей нет вообще — доступ открыт (bootstrap), пока не создан первый.
  - API: `seeding_api/auth.py` + `routers/auth.py` (`/auth/me`, CRUD ключей с защитой
    последнего admin), SSE учитывает ключ. Web: окно входа, гейтинг UI по роли, панель
    «Аккаунт» и управление ключами в Настройках (ключ показывается один раз).
- ✅ Публичный TLS: Caddy-фронт на :443 с локальным CA (`local_certs`, `default_sni` для
  доступа по IP). Оверрей `docker-compose.multi-engine.tls.yml`, подключается через
  `deploy-ct400.sh` (`DEPLOY_TLS=0` — отключить). HTTP→HTTPS редирект. Корневой CA
  экспортируется для доверия на клиентах. Гайд — `docs/TLS-AND-AUTH.md`.
- ✅ Учётки логин+пароль и сессии поверх ролей: таблицы `users` (PBKDF2-хеш пароля) и
  `user_sessions` (токен `ses_…`, хранится SHA-256, TTL `SEEDING_SESSION_TTL_HOURS`,
  по умолчанию 30 дней). `POST /auth/login` выдаёт токен (используется как обычный ключ),
  `POST /auth/logout` гасит сессию. Admin управляет пользователями (`/auth/users` CRUD,
  смена роли/пароля/блокировка с инвалидацией сессий, защита последнего администратора —
  учитываются и ключи, и пользователи). Web: окно входа по имени/паролю (с запасным входом
  по API-ключу), панель «Пользователи» в Настройках, кнопка «Выйти».
- ✅ Мягкий режим обслуживания на время restore БД: `seeding_api/maintenance.py` + middleware
  возвращает 503 на рабочие маршруты, пока идёт `pg_restore` (здоровье/вход/бэкапы доступны),
  флаг с авто-TTL. UI показывает понятное сообщение.
- ✅ Аудит-лог действий: таблица `audit_log`, middleware пишет все изменяющие запросы
  (POST/PUT/PATCH/DELETE) и входы — актор, роль, метод/путь, статус, IP и человекочитаемая
  сводка (`seeding_api/audit.py`). Хартбиты движков (`/engines/register`) исключены, есть
  авто-ретеншен (`SEEDING_AUDIT_RETENTION_DAYS`, по умолчанию 90 дней). Просмотр — `GET
  /api/v1/audit` (admin) и панель «Журнал действий» в Настройках.
- ✅ Постоянные лимиты трафика по движку: колонки `download_limit`/`upload_limit` в
  `engines` (миграция 0008), `POST /api/v1/engines/{id}/limits` сохраняет и сразу применяет
  лимит сессии libtorrent. Переживают перезапуск движка — переприменяются при
  саморегистрации. UI: панель «Лимиты движков» в Настройках (operator+).
  - ✅ Кумулятивные квоты по объёму на метку. Воркер (`sync_runtime_to_db`) копит отданные
    байты по дельтам `total_uploaded` каждого торрента метки (переживает рестарт движка —
    при сбросе счётчика libtorrent дельта берётся от нуля). При достижении лимита активные
    раздачи метки автоматически ставятся на паузу; кнопка «Сбросить» обнуляет счётчик и
    возобновляет приостановленные раздачи. Таблицы `label_quotas` + `torrent_meter`
    (миграция 0009), API `/api/v1/quotas` (CRUD + reset), панель «Квоты по меткам» в настройках.

- ✅ Приватные трекеры: для приватных раздач отключаются DHT/PEX/LSD (per-torrent флаги
  libtorrent `disable_dht|disable_pex|disable_lsd`). Приватность определяется по флагу
  `private` в метаданных **или** по эвристике passkey в адресе трекера (`passkey=`,
  `authkey=`, `torrent_pass=`, `secret=`, `/rss/`, `/announce.php`) — это покрывает ру-трекеры
  (rudub и т.п.), которые не ставят флаг private. Автоопределение применяется при добавлении
  и восстановлении (флаги пишутся в fastresume и переживают рестарт). API:
  `POST /api/v1/torrents/{id}/private` (`enabled: true|false|null`-авто) и
  `POST /api/v1/torrents/maintenance/reapply-private` (прогон по всем). UI: карточка
  «Приватный режим» в деталях (вкл/выкл/авто) + бейдж «🔒 Приватная», кнопка массового
  применения в Настройках. Внутренний эндпоинт движка `…/private`, поле `private` в снапшоте.

- ✅ Глобальная политика поиска пиров (DHT/PEX/LSD). Настройки в БД (`app_settings`,
  ключ `net_policy`, миграция 0011), применяются ко всем движкам сразу и переприменяются
  при саморегистрации движка (движок стартует с дефолтов из env). DHT/LSD — настройки
  сессии libtorrent (`enable_dht`/`enable_lsd`); PEX в lt 2.0 не имеет переключателя сессии,
  поэтому эмулируется per-torrent флагом `disable_pex` на всех раздачах. Приватные раздачи
  остаются без DHT/PEX/LSD независимо от глобальной политики. API `GET/POST
  /api/v1/settings/net`, внутренний `GET/POST /session/net-settings` у движка. UI: панель
  «Поиск пиров (глобально)» в Настройках (operator+).
