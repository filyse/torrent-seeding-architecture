# Доступ: авторизация (роли) и HTTPS (Фаза 5)

## Роли и API-ключи

Доступ к платформе — по именованным API-ключам с ролями:

| Роль | Что можно |
|------|-----------|
| `viewer` | только чтение (список раздач, детали, статус сервисов) |
| `operator` | всё чтение + управление раздачами (добавить/пауза/старт/удалить/лимиты/перенос) |
| `admin` | всё + управление API-ключами |

Проверка идёт по HTTP-методу: безопасные запросы (GET) требуют `viewer`, изменяющие
(POST/PATCH/DELETE) — `operator`. Управление ключами — `admin`.

### Хранение и совместимость
- В БД (`api_keys`) хранится **только SHA-256** ключа и короткий префикс — сам ключ
  показывается один раз при создании.
- Ключи из переменной `SEEDING_API_KEYS` (через запятую) считаются `admin` — аварийный
  доступ/бутстрап.
- Если ключей нет вообще (ни env, ни в БД) — доступ **открыт как admin**, чтобы можно было
  создать первый ключ. Как только появляется первый ключ — включается проверка.

### Первый вход
1. Открой веб-интерфейс. Пока ключей нет — доступ открыт.
2. Настройки → «API-ключи и доступ» → создай ключ с ролью `admin`, сохрани его.
3. Дальше интерфейс попросит ключ — введи его в окне входа (хранится в localStorage браузера).
4. Создай ключи `operator`/`viewer` для других людей/машин по необходимости.

### Программный доступ
Передавай ключ заголовком `X-API-Key: <ключ>` (для SSE — `?api_key=<ключ>`).

```bash
curl -H "X-API-Key: sk_..." https://<host>/api/v1/torrents
```

## HTTPS (Caddy, локальный CA)

Перед `web`/`api` стоит Caddy и терминирует TLS на `:443`. Сертификат выпускает
**локальный CA самого Caddy** (`local_certs`) — без интернета/ACME, подходит для доступа
по IP в локальной сети. HTTP (`:80`) редиректит на HTTPS. Старый вход по `:3000` (HTTP)
остаётся как запасной.

### Включение/отключение
Оверрей `docker-compose.multi-engine.tls.yml` подключается автоматически через
`scripts/deploy-ct400.sh`. Чтобы развернуть без TLS-фронта: `DEPLOY_TLS=0 scripts/deploy-ct400.sh up -d`.
Адрес сайта задаётся переменной `SEEDING_SITE` (по умолчанию `192.168.1.101`).

### Доверие сертификату на клиенте
Браузер будет предупреждать, пока не доверишь корневому CA Caddy. Экспорт корня:

```bash
docker cp containerd-caddy-1:/data/caddy/pki/authorities/local/root.crt ./caddy-root.crt
```

- **Windows**: `certutil -addstore -user Root caddy-root.crt` (или импорт в «Доверенные
  корневые центры сертификации»).
- **Linux**: скопировать в `/usr/local/share/ca-certificates/` и `update-ca-certificates`.
- **macOS**: добавить в Keychain и пометить как доверенный.

После этого `https://<SEEDING_SITE>/` открывается без предупреждений.
