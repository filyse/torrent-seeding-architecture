# Контракты между модулями

Цель: жёсткие границы, чтобы модули можно было разрабатывать и тестировать отдельно.

## Публичная граница

- **Браузер / десктоп** ↔ только **`api`** (HTTPS или HTTP в dev), пути вида `/api/v1/...`. Торренты: CRUD-образно список/создать/детали/пауза/возобновить/**удалить** (`DELETE /api/v1/torrents/{id}` — БД + снятие с движка). Опционально **`X-API-Key`**, если задано `SEEDING_API_KEYS` на API.
- **`api`** ↔ **`engine`**: внутренняя сеть Docker (базовый URL `ENGINE_URL`, порт по умолчанию `8081`). Префикс внутреннего REST: **`/internal/v1`**. При старте API (lifespan) выполняется **восстановление рантайма**: строки БД со статусами `downloading` / `seeding` / `paused` и валидным `magnet_uri` снова регистрируются в движке; отключение: `SEEDING_ENGINE_RESTORE=0`.

  | Метод | Путь | Назначение |
  |-------|------|------------|
  | GET | `/health` | Health вне префикса (удобно для compose) |
  | GET | `/internal/v1/health` | Дубль health |
  | GET | `/internal/v1/torrents` | Список рантайм-хендлов в движке |
  | POST | `/internal/v1/torrents` | Регистрация: тело `{ "db_id", "magnet_uri?", "save_path" }` |
  | GET | `/internal/v1/torrents/{db_id}` | Снимок рантайма |
  | POST | `/internal/v1/torrents/{db_id}/pause` | Пауза в движке |
  | POST | `/internal/v1/torrents/{db_id}/resume` | Возобновление |
  | DELETE | `/internal/v1/torrents/{db_id}` | Убрать торрент из рантайма движка (не удаляет строку в БД) |

  Публичный OpenAPI движка не обязателен для браузера; контракт синхронизируют агенты **4** и **5**.
- **`api`** ↔ **`db`**: только через слой репозиториев из пакета `seeding_db` (каталог `db/`).
- **`api`** ↔ **`queue`**: постановка задач через пул Redis (enqueue); тела задач — сериализуемые DTO, не «сырые» ORM-объекты. Публичные эндпоинты (префикс в `main.py`: `/api/v1/jobs/…`; при `SEEDING_API_KEYS` — заголовок `X-API-Key`):

  | Метод | Путь | Назначение |
  |-------|------|------------|
  | POST | `/api/v1/jobs/noop` | Задача-заглушка `noop_report` |
  | POST | `/api/v1/jobs/engine-health-check` | Задача `check_engine_health` → `GET ENGINE_URL/health` |

- **`queue` workers** ↔ **`db`**: по текущему коду воркер может вызывать **`engine`** по HTTP (`check_engine_health`); запись в БД из воркера — по мере появления задач, через те же модели `seeding_db`.

## Данные на диске

- Файлы контента и каталог для `.torrent` монтируются в **`engine`** (и при необходимости read-only в `api` для выдачи метаданных — только если явно нужно; по умолчанию пути хранятся в БД, читает движок).

## База данных

- В Docker с **несколькими контейнерами** предпочтительно **PostgreSQL** (общий том или внешний инстанс). **SQLite** допустим для однопроцессного dev или если только один сервис пишет в БД (иначе — блокировки и сеть).
- Миграции ведёт агент **БД**; остальные не правят схему в обход Alembic.

## Версионирование API

- Внешнее API: префикс `/api/v1/`. Ломающие изменения → `v2` или согласование с координатором.
