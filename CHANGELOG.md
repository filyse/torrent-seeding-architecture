# Журнал изменений

Учёт фиксов и улучшений по версиям. Версии компонентов независимы:
`web` (`web/src/version.ts`, `WEB_VERSION`) и ядро `api`/`engine`/`queue` (`__version__`).
Бамп — вручную при заметных изменениях; дата сборки пишется в образ (`/BUILD_TIME`).

Формат: `## <компонент> <версия> — <дата>` и список изменений.
Свежие записи — сверху.

## engine 1.1.4 — 2026-07-21

- **engine: быстрый листинг каталогов на HDD (`/fs/browse`).** Раньше `browse()` вызывал
  `entry.stat()` для КАЖДОГО элемента, включая каталоги. На HDD-движках (b\*) с холодным
  кэшем это тысячи random-seek к inode — на каталоге с ~1700 записей (напр. b1) выбор
  контента «подвисал» на секунды; на a\* (SSD/NVMe) незаметно. Теперь `stat()` делается
  только для файлов (их размер нужен UI), а для каталогов пропускается: `size=0`, поле
  `modified` UI не использует. Листинг папок сводится к одному `readdir` — мгновенно даже
  на HDD. FTP-листинг тормозит по той же причине (LIST тоже делает stat на элемент), но это
  вне нашего кода — свойство диска/FTP-сервера.

## engine 1.1.2 + api 1.5.2 — 2026-07-18

Производительность и «ровность» создания торрентов на HDD-движках (b*), где хеширование
конкурирует за диск с сотнями активных раздач.

- **engine: хеширование вынесено в отдельный процесс** (`multiprocessing`, start method
  `spawn`). Раньше `libtorrent.set_piece_hashes` держал GIL в рабочем потоке и на 2–3с
  подвешивал событийный цикл движка — из-за чего опрос статуса тормозил, а под нагрузкой
  запросы оркестратора падали в таймаут («engine unavailable»). Теперь вся CPU/IO-работа
  идёт в дочернем процессе, а поток-надзиратель лишь качает прогресс через очередь
  (`_hash_worker`, `CreateStatus`-очередь, `_MP_CTX`). Событийный цикл свободен → прогресс
  в UI обновляется плавно, здоровье/опрос движка не деградируют во время создания.
- **engine: число одновременных задач создания — env `SEEDING_CREATOR_WORKERS`, дефолт 1.**
  На HDD параллельное хеширование двух папок вызывает seek-thrashing и замедляет обе;
  последовательная обработка (1 за раз) быстрее суммарно и идёт ровнее. На SSD/NVMe-движках
  можно поднять (`SEEDING_CREATOR_WORKERS=2+`).
- **api: очередь создания опрашивается параллельно** (`GET /api/v1/creator/tasks`,
  `asyncio.gather`) — один занятый HDD-движок больше не тормозит обновление всей очереди в UI.
- Флот пересобран на `engine 1.1.2` (a1–a3, b1–b6), оркестратор — `api 1.5.2`.

## api 1.4.2 — 2026-06-21

- **Фикс 500 на странице раздачи (`GET /torrents/{id}`).** Если одна и та же раздача
  крутится на нескольких движках (напр. после переноса осталась копия), её `info_hash`
  уже занят другой записью. Эндпоинт пытался записать `info_hash` в запись с `NULL` и
  падал с `IntegrityError` (`UniqueViolation` по `torrents_info_hash_key`) → 500. Теперь
  перед записью проверяется наличие записи с таким `info_hash` (`get_by_info_hash`);
  при дубликате запись не выполняется, страница открывается нормально.

## engine 1.0.1 + api 1.4.1 — 2026-06-21

- **Верхняя панель: «Скачивание» показывает только полезные данные (payload).** Раньше
  агрегат брал `download_rate`/`upload_rate` из `session_stats` libtorrent — это весь
  входящий трафик вместе с протокольным оверхедом (DHT, трекеры, запросы кусков, uTP-ACK,
  ip-overhead). При чистой раздаче это давало «фантомные» ~2 МБ/с «скачивания», хотя ничего
  не качается. Теперь движок отдаёт `payload_download_rate`/`payload_upload_rate` (реальные
  данные раздач), с откатом на total только если биндинг не отдаёт payload-поля. При полной
  раздаче «Скачивание» ≈ 0 (engine `torrent_runtime.py`).
- **Верхние карточки обновляются вживую (канал `stats`).** Добавлен лёгкий пуш агрегатов
  сессии в WS-канал `stats` каждые ~4с из `ws_pollers` (та же форма, что у SSE `/stream` и
  `GET /session/stats`). Раньше агрегаты публиковал только тяжёлый цикл полного снапшота
  рантайма, который на больших инсталляциях шёл дольше интервала — карточки «Раздач /
  Скачивание / Отдача» подвисали. Дублирующий пуш из `runtime_snapshot` убран.
- **Фикс молчаливого слома WS-авторизации под uvicorn+wsproto.** Предложенные клиентом
  сабпротоколы (`bearer.<key>`) читаются из ASGI-`scope`, а не из заголовка
  `sec-websocket-protocol` (под wsproto его в `scope["headers"]` может не быть). Из-за этого
  WS уходил в анонимную ветку → `4401` → бесконечный reconnect, и верхняя панель не
  обновлялась. Заголовок оставлен как фолбэк (`routers/ws.py`).
- **Флот приведён к одной версии.** Все 9 движков (a1–a3, b1–b6) пересобраны на
  `engine 1.0.1` с payload-метрикой.
