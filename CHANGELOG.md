# Журнал изменений

Учёт фиксов и улучшений по версиям. Версии компонентов независимы:
`web` (`web/src/version.ts`, `WEB_VERSION`) и ядро `api`/`engine`/`queue` (`__version__`).
Бамп — вручную при заметных изменениях; дата сборки пишется в образ (`/BUILD_TIME`).

Формат: `## <компонент> <версия> — <дата>` и список изменений.
Свежие записи — сверху.

---

## engine 1.1.3 + api 1.5.3 + web 1.7.2 — 2026-07-19

Управление жизненным циклом задач создания.

- **web + api + engine: удаление задачи из «Очереди создания».** В строке задачи —
  кнопка «Удалить» (с подтверждением): убирает задачу из очереди и её `.torrent` из
  памяти движка; активную задачу при этом прерывает (останавливает дочерний процесс
  хеширования). Реализация: `DELETE /internal/v1/creator/tasks/{id}`
  (`CreatorService.delete`), проксирование `DELETE /api/v1/creator/tasks/{engine}/{id}`.
- **engine: автоочистка задач по TTL — 24 часа.** Задача (вместе с `.torrent` в памяти)
  живёт не дольше `SEEDING_CREATOR_TASK_TTL` (сек, дефолт 86400) и автоудаляется:
  фоновый reaper-поток + прунинг при обращении к очереди. Освобождает память от
  несозданных/несохранённых торрентов без перезапуска движка.
- Флот пересобран на `engine 1.1.3` (a1–a3, b1–b6), оркестратор — `api 1.5.3`,
  web — 1.7.2. `queue` — 1.0.0.

## engine 1.1.2 + api 1.5.2 — 2026-07-18

Производительность и «ровность» создания торрентов на HDD-движках (b*), где хеширование
конкурирует за диск с сотнями активных раздач.

- **engine: хеширование вынесено в отдельный процесс** (`multiprocessing`, start method
  `spawn`). Раньше `libtorrent.set_piece_hashes` держал GIL в рабочем потоке и на 2–3с
  подвешивал событийный цикл движка — из-за чего опрос статуса тормозил, а под нагрузкой
  запросы оркестратора падали в таймаут («engine unavailable»). Теперь вся CPU/IO-работа
  идёт в дочернем процессе, а поток-надзиратель лишь качает прогресс через очередь
  (`_hash_worker`, `CreateStatus`-очередь, `_MP_CTX`). Событийный цикл свободен → прогресс
  в UI обновляется плавно, здоровье/опрос движка не деградируют во время создания.
- **engine: число одновременных задач создания — env `SEEDING_CREATOR_WORKERS`, дефолт 1.**
  На HDD параллельное хеширование двух папок вызывает seek-thrashing и замедляет обе;
  последовательная обработка (1 за раз) быстрее суммарно и идёт ровнее. На SSD/NVMe-движках
  можно поднять (`SEEDING_CREATOR_WORKERS=2+`).
- **api: очередь создания опрашивается параллельно** (`GET /api/v1/creator/tasks`,
  `asyncio.gather`) — один занятый HDD-движок больше не тормозит обновление всей очереди в UI.
- Флот пересобран на `engine 1.1.2` (a1–a3, b1–b6), оркестратор — `api 1.5.2`.
  `web` без изменений (остаётся 1.7.1), `queue` — 1.0.0.

## engine 1.1.1 + api 1.5.1 + web 1.7.1 — 2026-07-18

Правки UX создания торрентов и «Очередь создания».

- **web: браузер сразу открывает папку контента движка.** При выборе движка вместо
  корня `/data` (со служебными `.fastresume`/`.state`/`.torrents`) открывается его
  подпапка контента (`basename(storage_prefix)`, напр. `/data/a1` → `a1`); при
  недоступности — фолбэк на корень.
- **web: режим по умолчанию — «Только создать (.torrent)».** Поля «Метка» и «Название»
  показываются только для режима «Создать и раздавать».
- **web: ручное скачивание вместо авто.** В режиме «только создать» `.torrent` больше не
  скачивается автоматически — в строке задачи появляется кнопка «Скачать».
- **web + api + engine: «Очередь создания».** Кнопка в тулбаре открывает список задач
  создания со ВСЕХ движков с действиями Скачать / Раздать / Отмена и опросом активных.
  Позволяет вернуться к задачам после закрытия вкладки браузера (пока движок не
  перезапущен). Реализация: движок отдаёт список задач (`GET /internal/v1/creator/tasks`,
  `CreatorService.list_all`), оркестратор агрегирует по всем движкам
  (`GET /api/v1/creator/tasks`). **Redis/`queue_worker` не задействованы** — задачи
  создания живут в памяти движка (эфемерно), поэтому версия `queue` не меняется.
- **web: кнопка «Создать торрент»** получила иконку и единый со «соседями» размер/стиль.
- Флот пересобран на `engine 1.1.1` (a1–a3, b1–b6).

## engine 1.1.0 + api 1.5.0 + web 1.7.0 — 2026-07-18

- **Создание торрентов из контента на дисках движков (RelaySeed «creator»).** Логика
  портирована из внешнего torrent_api (libtorrent 1.2) под libtorrent 2.0 (v1-only) и
  встроена прямо в движки — при развёртывании нового движка функция доступна сразу.
  - Движок (`engine/seeding_engine/creator.py` + внутренний API): обзор диска
    (`/internal/v1/fs/browse`), постановка задачи, статус/прогресс, отмена, выдача байтов
    `.torrent` (`/internal/v1/creator/tasks…`). 16 МБ piece size, проверка
    последовательности серий (по умолчанию выключена), защита от path traversal.
  - Оркестратор (`api/seeding_api/routers/creator.py`): проксирование к движку + два
    режима — «только создать» (эфемерный `.torrent` стримится в браузер) и «создать и
    раздавать» (созданный торрент сразу ставится на сидинг через существующий pipeline).
  - Web: модал «Создать торрент» (выбор движка/папок, мультивыбор, режимы, прогресс).
  - Примечание: фича изначально выкатывалась в прод без бампа версий (движок оставался
    `1.0.3`) — учёт восстановлен этой записью.

## api 1.4.2 — 2026-06-21

- **Фикс 500 на странице раздачи (`GET /torrents/{id}`).** Если одна и та же раздача
  крутится на нескольких движках (напр. после переноса осталась копия), её `info_hash`
  уже занят другой записью. Эндпоинт пытался записать `info_hash` в запись с `NULL` и
  падал с `IntegrityError` (`UniqueViolation` по `torrents_info_hash_key`) → 500. Теперь
  перед записью проверяется наличие записи с таким `info_hash` (`get_by_info_hash`);
  при дубликате запись не выполняется, страница открывается нормально.

## engine 1.0.3 — 2026-06-21

- **Фикс владельца контента 1000:1000 для ОСТАЛЬНЫХ путей переноса.** В 1.0.2 chown был
  добавлен только в `import_local` и `import_remote`, но прямой перенос движок→движок
  (`import_direct`, скачивание контента напрямую у движка-источника по TLS) и возобновляемый
  пофайловый приём (`import_finalize`) писали контент от root **без** chown — перенесённые так
  раздачи снова оказывались `root:root` и были недоступны по FTP. Теперь chown владельца
  (1000:1000) выставляется во всех четырёх путях импорта.

## engine 1.0.2 — 2026-06-21

- **Перенос между движками: владелец контента 1000:1000 (доступ по FTP).** Движок в
  контейнере работает от `root`, поэтому при импорте файлы/каталоги создавались от `root:root`
  (`extractall(filter="data")` и `copystat` не переносят владельца) — по FTP (uid 1000) к
  перенесённым раздачам не было доступа. Теперь после копирования (`import_local`) и распаковки
  tar-потока (`import_remote`) движок выставляет владельца перенесённого дерева в `1000:1000`
  (переопределяется через `SEEDING_CONTENT_UID` / `SEEDING_CONTENT_GID`; `-1` — не менять).
  Затрагивается только перенесённое дерево (top-level контента раздачи), не весь том.

## engine 1.0.1 + api 1.4.1 — 2026-06-21

- **Верхняя панель: «Скачивание» показывает только полезные данные (payload).** Раньше
  агрегат брал `download_rate`/`upload_rate` из `session_stats` libtorrent — это весь
  входящий трафик вместе с протокольным оверхедом (DHT, трекеры, запросы кусков, uTP-ACK,
  ip-overhead). При чистой раздаче это давало «фантомные» ~2 МБ/с «скачивания», хотя ничего
  не качается. Теперь движок отдаёт `payload_download_rate`/`payload_upload_rate` (реальные
  данные раздач), с откатом на total только если биндинг не отдаёт payload-поля. При полной
  раздаче «Скачивание» ≈ 0 (engine `torrent_runtime.py`).
- **Верхние карточки обновляются вживую (канал `stats`).** Добавлен лёгкий пуш агрегатов
  сессии в WS-канал `stats` каждые ~4с из `ws_pollers` (та же форма, что у SSE `/stream` и
  `GET /session/stats`). Раньше агрегаты публиковал только тяжёлый цикл полного снапшота
  рантайма, который на больших инсталляциях шёл дольше интервала — карточки «Раздач /
  Скачивание / Отдача» подвисали. Дублирующий пуш из `runtime_snapshot` убран.
- **Фикс молчаливого слома WS-авторизации под uvicorn+wsproto.** Предложенные клиентом
  сабпротоколы (`bearer.<key>`) читаются из ASGI-`scope`, а не из заголовка
  `sec-websocket-protocol` (под wsproto его в `scope["headers"]` может не быть). Из-за этого
  WS уходил в анонимную ветку → `4401` → бесконечный reconnect, и верхняя панель не
  обновлялась. Заголовок оставлен как фолбэк (`routers/ws.py`).
- **Флот приведён к одной версии.** Все 9 движков (a1–a3, b1–b6) пересобраны на
  `engine 1.0.1` с payload-метрикой.

## api 1.4.0 + web 1.5.0 — 2026-06-21

- **«Обновить торрент» — замена раздачи новым .torrent в один шаг.** Сценарий «в сезонный
  пак добавили новую серию»: больше не нужно искать старую раздачу в списке, удалять её
  руками, заново добавлять и выбирать движок/путь.
  - Новая кнопка «Обновить торрент» в шапке рядом с «Добавить торрент». Диалог: загрузка
    одного или нескольких новых `.torrent` (пакетно).
  - Авто-сопоставление по имени файла (`POST /api/v1/torrents/update/match`): для каждого
    файла показывается найденная старая раздача (движок · путь). При нескольких совпадениях —
    выбор из списка; если не найдено — ручной поиск раздачи прямо в диалоге.
  - Замена (`POST /api/v1/torrents/{id}/replace`): старая раздача снимается с раздачи **без
    удаления файлов** (`delete_files=False`), новый `.torrent` добавляется в **тот же путь на
    том же движке** с recheck — уже скачанные серии засчитываются, докачивается только новая,
    затем встаёт на раздачу. Движок, путь и метка наследуются автоматически.

## web 1.4.1 — 2026-06-21

- **Широкая раскладка для таблицы.** В табличном виде страница разворачивается почти
  на всю ширину окна (`max-width` контейнера снимается, до 118rem) — колонки больше не

- **Понятный переключатель вида.** Вместо двух отдельных иконок (вид + плотность) —

  сразу выбирает режим, активный сегмент подсвечен.

## web 1.4.0 — 2026-06-21

- **Табличный вид списка раздач (стиль ruTorrent).** В тулбаре списка — переключатель
  «карточки ↔ таблица» (иконка, выбор сохраняется в `localStorage`, ключ `ui.view`).
  Таблица: плотные строки с зеброй и подсветкой при наведении, липкая шапка, колонки
  Имя / Статус / Размер / Готово (мини-бар + %) / Отдано / Ратио / ↓ / ↑ / Сиды / Пиры /
  Осталось / Метка. Числовые колонки выровнены вправо (моноширинные цифры). Клик по
  заголовку сортирует (использует серверную сортировку: имя/размер/прогресс/отдано/ратио/
  ↓/↑/пиры), активная колонка отмечена индикатором. Чекбокс выбора в каждой строке +
  «выбрать все на странице» в шапке — работает общая панель массовых действий.
  Компактные кнопки в строке: пауза/старт и удаление. Переключатель плотности
  (плитки/строки) относится только к карточкам и в табличном виде скрыт.

## web 1.3.1 — 2026-06-21

- **Чистые URL без `#`.** Роутинг переведён с hash-routing на History API:
  адреса стали `…/torrent/123` и `…/settings` вместо `…/#/torrent/123`. На главной
  больше нет «висящего» `#`. Работают кнопки браузера назад/вперёд, прямые ссылки и
  перезагрузка на глубоком пути (nginx уже отдаёт `index.html` через
  `try_files … /index.html`). Ссылки (`href`) теперь указывают на реальные пути —
  работает средний клик / «открыть в новой вкладке».
