# Агенты и порядок работы

Координация ведётся через этот репозиторий: ветки, PR, чеклисты в `docs/reports/` (создаёт координатор или агент по шаблону).

## Роли

| № | Агент | Ответственность |
|---|--------|-----------------|
| **1** | **Координатор (главный)** | Приоритеты, контракты между модулями, приёмка PR, сборка `docker-compose`, релизные теги. Принимает отчёты от агентов **8** и **9** и решает блокеры. |
| **2** | **Десктоп (Windows)** | Клиент под Windows (рекомендуемый стек: Python + PyQt6/PySide6 или отдельно оговорённый). Настройка URL API, список торрентов, команды, отображение статуса. |
| **3** | **Веб** | SPA или лёгкий фронт (React/Vue/Svelte и т.д.): те же сценарии, что у десктопа, без прямого доступа к диску сервера. |
| **4** | **API** | FastAPI: эндпоинты, валидация, авторизация (если нужна), вызовы к движку и БД, CORS, OpenAPI. |
| **5** | **Движок** | Процесс libtorrent: сессия, add/remove/pause, метрики, устойчивый shutdown; канал связи с API (HTTP/gRPC/внутр. — см. `docs/INTEGRATION.md`). |
| **6** | **БД** | Модели, миграции (Alembic), репозитории/DAO, согласование с API и при необходимости с очередью. |
| **7** | **Очередь** | Redis, ARQ: определения задач, воркер, настройки; API только ставит задачи. |
| **8** | **QA / интеграционные тесты** | Сквозные сценарии (compose), API + движок + БД, регрессия контрактов. Отчёты координатору. |
| **9** | **QA / клиенты и нагрузка** | Тесты веб/десктоп против моков или staging API, базовые проверки UX и ошибок сети. Отчёты координатору. |

## Поток сдачи

1. Агент **2–7** открывает PR в свой каталог (или согласованные пути), не ломая границы из `docs/INTEGRATION.md`.
2. Перед запросом ревью: локальный чеклист из соответствующего раздела `docs/PLAN_BY_AGENT.md`.
3. Агенты **8** и **9** прогоняют тесты, фиксируют результат в `docs/reports/YYYY-MM-DD-<agent>.md` (шаблон ниже).
4. **Координатор** смотрит PR + отчёты QA, мержит или возвращает на доработку.

## Шаблон отчёта QA (файл в `docs/reports/`)

```markdown
# Отчёт: <дата> — агент 8 или 9

## Область
- Ветка / коммит:
- Затронутые модули:

## Прогон
- [ ] unit / integration команды и результат
- [ ] ручные шаги (если есть)

## Найденные дефекты
| ID | Критичность | Описание | Модуль |
|----|-------------|----------|--------|

## Вердикт
Готово к мерже / Не готово — причина:
```

## Запреты

- Клиенты не подключаются к Redis, БД и внутренним портам движка напрямую.
- Движок не содержит публичного REST для браузера без слоя API (если иначе не зафиксировано в `INTEGRATION.md`).

## Защита публичного API (опционально)

Если на стенде задано `SEEDING_API_KEYS`, клиенты передают **`X-API-Key`** (веб: `localStorage.seedingApiKey`, CLI: `--api-key`). Подробности — [`api/README.md`](api/README.md).
